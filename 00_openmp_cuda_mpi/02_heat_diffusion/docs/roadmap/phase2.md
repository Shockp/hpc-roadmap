### Phase 2: Multicore Scaling (OpenMP) Specification

This document details the architectural requirements and optimization strategies for the Phase 2 OpenMP implementation. Building upon the Phase 1 C++ baseline, this iteration introduces shared-memory parallelism. To maintain our high-performance HPC base, all code must continue to strictly follow the Google C++ Style Guide, ensuring thread-safe, readable, and highly optimized routines.

---

### 1. Core Objectives
* **Shared-Memory Parallelism:** Distribute the computational load of the $N \times N$ matrix across multiple CPU cores using OpenMP compiler directives.
* **Overhead Minimization:** Structure parallel regions to minimize the penalty of thread creation and synchronization barriers.
* **Empirical Analysis:** Systematically test and document the performance impact of different loop scheduling strategies.

### 2. Parallelization Strategy
The primary target for acceleration is the nested loop computing the 5-point stencil. 

* **Target Identification:** Apply `#pragma omp parallel for` to the outermost spatial loop (the row iterator) to distribute chunks of rows among available threads.
* **Data Environment Clauses:** Explicitly declare shared and private variables using `shared(T_old, T_new, N)` and `private(i, j)` to prevent race conditions.
* **Memory Access Patterns:** Because C++ uses row-major order, assigning complete rows to threads ensures contiguous memory access and maximizes L1/L2 cache utilization.

### 3. Loop Scheduling Analysis
The project requires a formal analysis of OpenMP scheduling clauses. You must implement and benchmark at least two strategies to prove your understanding of workload distribution.

* **Static Scheduling (`schedule(static)`):** Divides the loop iterations into equal-sized chunks assigned to threads at compile-time. This is generally optimal for our grid because the computational cost per stencil point is identical and predictable.
* **Dynamic Scheduling (`schedule(dynamic)`):** Assigns chunks to threads at runtime as they become available. You must measure the overhead introduced by this dynamic assignment and explain why it is likely less efficient for a uniform heat diffusion workload.

### 4. Advanced Thread Optimization
To achieve true HPC performance, simply adding pragmas is not enough; the architecture must actively avoid common multithreading bottlenecks.

* **Thread Lifecycle Management:** Avoid opening and closing the parallel region inside the time-step loop. Instead, declare `#pragma omp parallel` outside the time loop, and use `#pragma omp for` inside, utilizing `#pragma omp barrier` to synchronize threads between the `T_old` read and `T_new` write phases.
* **False Sharing Prevention:** Ensure that the chunk size assigned to each thread is a multiple of the CPU cache line size (typically 64 bytes). If threads write to adjacent memory addresses residing on the same cache line, it triggers severe performance penalties due to cache invalidation.

### 5. Validation & Benchmarking Protocol
The OpenMP implementation must be rigorously tested against the Phase 1 baseline.

* **Correctness Verification:** Execute the code on the $64 \times 64$ grid and programmatically assert that the output matrix exactly matches the output generated by the sequential baseline.
* **Scalability Testing:** Run the simulation on the hardware stress-test size ($4096 \times 4096$).
* **Thread Scaling:** Measure the execution time using 2, 4, 8, and the maximum number of physical cores available on your testing machine.
* **Speedup Calculation:** Compute the strong scaling speedup factor ($S = \frac{T_{sequential}}{T_{parallel}}$) for each thread count and document the parallel efficiency.
